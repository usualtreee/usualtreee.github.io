二、模型优化技术全景

图优化与 torch.compile

  • 图优化（Graph Optimization）：重排计算图结构，合并算子、常量折叠、死代码消除。训练框架和部署编译器都做
  • torch.compile：PyTorch 2.0 特性，把 Python 模型编译成高效底层代码，自动算子融合、生成优化 CUDA Kernel。主要加速训练，也可用于推理
  • 与量化的关系：图优化改"怎么算"，量化改"用什么精度算"。两者是独立维度，转换时通常一起做

除量化外的优化方法

优化手段 说明 边缘部署适用性
算子融合 Conv+BN+ReLU 合并，减少访存和 Kernel Launch ⭐⭐⭐ 通用
剪枝 非结构化/结构化，删掉不重要连接 ⭐⭐ 需训练资源
知识蒸馏 小模型模仿大模型输出分布 ⭐⭐ 需训练资源
内存优化 内存池、零拷贝、对齐、激活复用 ⭐⭐⭐ 边缘核心
编译器优化 AutoTVM、Kernel Auto-Tuning、Layout 转换 ⭐⭐⭐ TensorRT 自动做
并行与流水线 多线程、多 Stream、Continuous Batching ⭐⭐⭐ 边缘核心
缓存/Tiling 分块计算，数据留在片上 SRAM 复用 ⭐⭐⭐ NPU/GPU 核心
架构设计 MobileNet、RepVGG、Mamba 等轻量结构 ⭐⭐ 训练阶段决定
动态优化 Early Exit、Dynamic Shape、Adaptive Batch ⭐⭐ 部分场景
低秩分解 SVD、权重共享、哈希 trick ⭐ 较少用

边缘部署重点：算子融合 + 内存优化 + 并行流水线 + Tiling。剪枝/蒸馏/架构设计需要训练资源，边缘团队通常做不了。

三、RKNN vs NVIDIA 量化深度对比

核心结论

  • RKNN：几乎只有 PTQ，QAT 极弱且不稳定，调试工具几乎为零。量化是黑盒
  • TensorRT：PTQ + QAT + Explicit Quantization 全栈开放，逐层可控

RKNN 黑盒环节

  • 图优化策略不可见
  • 校准算法固定（normal/mmse），不可自定义
  • 每层 scale/zero_point 不可查看
  • 算子映射与调度不可干预
  • 内存管理策略不可见
  • 精度崩了只能换数据集或祈祷

NVIDIA 开放环节

  • polygraphy 逐层对比 ONNX vs TensorRT 输出
  • 自定义 IInt8Calibrator
  • 逐层设精度（FP32/FP16/INT8）
  • 手写 TensorRT Plugin / CUDA Kernel 替换任意算子
  • Nsight Systems / Profiler 完整性能分析

共通点极少

  • 只有"PTQ 概念"和"校准数据集"是共通的
  • 具体实现、调试、控制手段完全不同
  • RK 量化崩了只能换数据集祈祷；TensorRT 能逐层定位、局部回退、写 Plugin 修

四、硬件与底层认知

NPU vs GPU 架构差异

  • GPU：通用并行（SIMT），CUDA Core/Tensor Core，完全可编程，功耗高（几百瓦）
  • NPU：专用 DSA，MAC 乘加阵列 + 大容量片上 SRAM，硬连线控制，不可编程，功耗极低（几瓦~十几瓦）
  • 性能瓶颈都是内存墙：优化终极敌人是访存带宽，不是纯算力

华为昇腾达芬奇架构

  • Scalar Unit：标量控制，管循环/分支/地址/调度
  • Vector Unit：SIMD 向量运算，逐元素操作（ReLU/Add/Norm）
  • Cube Unit：矩阵乘加，专门干 Conv/FC

MAC 与算力

  • 1 MAC = 1 次乘法 + 1 次加法
  • 算力换算：MAC 数 × 核数 × 频率 × 2 = TOPS（×2 因为 1 MAC = 2 次基本运算）

GaN 器件定位

  • GaN 是功率半导体，适合做电源开关（数据中心 PSU、机器人关节驱动、车载 OBC）
  • 不适合做数字 MAC/AI 计算：工艺节点微米级、功耗高、密度低、无 EDA 生态
  • 抗辐射 FPGA（Xilinx XQR、Microchip RTG4）已覆盖绝大多数恶劣环境数字控制需求
  • 蹭 AI 的正确姿势：给 AI 芯片/数据中心做电源优化，而不是替代计算芯片

M5StickS3

  • ESP32-S3 物联网开发板，拇指大小，0.85 英寸 LCD，集成 IMU
  • 可跑 TensorFlow Lite Micro 轻量模型（语音唤醒、手势识别）
  • 和用户路线无关：超低功耗 IoT 玩具，非工业级边缘 AI

RK3588 SoC 与开发板

RK3588 SoC 配置：

  • CPU：四核 A76（2.4GHz）+ 四核 A55（1.8GHz）
  • GPU：ARM Mali-G610 MC4
  • NPU：瑞芯微自研，三核，6 TOPS INT8，支持 INT4/INT8/FP16/BF16/TF32
  • 内存：LPDDR4/4X/5，最高 32GB（开发板常见 4/8/16GB）
  • 视频：8K@60fps 解码，8K@30fps 编码，支持多路 4K 同时解码
  • 显示：HDMI 2.1（8K）、MIPI DSI、eDP、DP 1.4，支持三屏异显
  • 网络：双 RGMII（可扩展双千兆/万兆）
  • 接口：PCIe 3.0、USB 3.1、SATA 3.0、CAN-FD

迅为/正点原子开发板差异：

  • SoC 核心（RK3588）完全一样
  • 差异在：内存容量、散热设计、MIPI CSI 路数、底板接口引出、系统镜像适配
  • 对 4 路视频项目：建议内存 ≥8GB（16GB 更稳），必须加风扇散热

六、多线程与同步机制

notify_one vs notify_all

  • notify_one：只唤醒一个等待的线程。适用于单消费者或线程池取一个任务
  • notify_all：唤醒所有等待的线程。适用于多路独立队列、状态变更（退出/flush）、多消费者竞争
  • 4 路视频项目建议：如果每路独立条件变量，用 notify_one；如果共享条件变量，用 notify_all 避免"唤醒丢失"

C++ vs FreeRTOS

  • std::condition_variable 是 C++11 标准库，依赖 Linux 完整操作系统
  • MCU / FreeRTOS 不用这个：资源有限，用 C 语言 + FreeRTOS API
  • FreeRTOS 等价机制：
    • 二值信号量（Binary Semaphore）≈ condition_variable + mutex
    • 计数信号量 ≈ 多资源场景
    • 任务通知（Task Notifications）≈ 最轻量通知
    • 队列（Queue）≈ 自带同步的生产者-消费者

七、其他技术点

文件类型

  • .sh：Shell 脚本，Linux 可执行文本文件，由 bash/sh 解释执行
  • .so：用户态动态链接库（Shared Object），封装函数/变量/数据
  • .ko：内核模块（Kernel Object），直接操作硬件寄存器

GPIO 驱动逻辑

  • gpio_export：向内核申请暴露 GPIO 引脚到 sysfs
  • gpio_unexport：释放 GPIO 引脚
  • 代码逻辑纠正：如果 GPIO 已存在（已导出），不应 unexport，应直接使用；程序退出前才 unexport 释放资源

IP 公司与 EDA 公司

  • EDA 公司（Cadence、Synopsys、Mentor）：卖芯片设计工具（画电路、仿真、验证、布局布线）
  • IP 公司（ARM、芯来科技 Nuclei、Imagination、CEVA）：卖电路图纸（CPU/GPU/DSP 核授权）
  • Cadence/Synopsys 也卖 IP（如 Synopsys DesignWare DDR PHY），但主业是 EDA 工具

SoC / 核心板 / 开发板 层级

  • SoC：芯片（瑞芯微设计 RK3588，台积电/三星代工）
  • 核心板：焊了 SoC + 内存 + 存储 + 电源的小 PCB
  • 开发板：核心板插在上面，引出 HDMI/USB/网口/摄像头等接口的大 PCB
