雅思听力学习 Q&A 记录

日期：2026-04-22
来源：剑桥雅思 20 Test 3 Part 4 “Inclusive Design”

❓ Q1：metabolic 是什么意思？

✅ 解答：

  • 形容词，中文意为 “新陈代谢的”、“代谢的”。
  • 常见搭配：
    • metabolic rate：代谢率
    • metabolic disease：代谢疾病

❓ Q2：become aware of 是什么意思？

✅ 解答：

  • 短语，意为 “意识到、察觉到”。
  • 强调从“不知道”到“知道”的状态转变。
  • 例句：Designers have become aware of the need to be inclusive.（设计师意识到了包容性的必要性。）

❓ Q3：This still isn’t always the case 怎么理解？

✅ 解答：

  • 意为 “情况依然并非总是如此” 或 “这尚未成为普遍现象”。
  • 在听力原文中，它是一个 转折信号，指出前文提到的积极例子（可调节桌子）虽然常见，但并未完全普及。

❓ Q4：scenario 是什么意思？

✅ 解答：

  • 名词，意为 “场景”、“情形”、“局面”。
  • 常见搭配：
    • worst-case scenario：最坏的情况
    • best-case scenario：最好的情况
    • in this scenario：在这种情况下

📌 备注

  • 以上词汇均出现在雅思听力学术类 Part 4 场景中，建议结合原文语境记忆。

Linux 驱动学习笔记 - 2026-04-22

一、module_init 是如何运行的？

1.1 两种运行路径

module_init 宏指定的初始化函数，根据驱动编译方式的不同，走两条完全不同的执行路径：

编译方式 执行路径 触发时机
编译为模块 (.ko) insmod → 系统调用 → 内核模块加载器 → 直接调用 module_init 指定的函数 手动加载模块时
编译进内核 (built-in) module_init → 演变为 device_initcall → 函数指针放入特定段 → 内核启动时 do_initcalls() 统一遍历调用 内核启动过程中

1.2 “放到对应的段里面”是什么意思？

  • 段（Section）：编译器将代码和数据归类打包的单元（如 .text 放代码，.data 放已初始化全局变量）。
  • 内核的自定义段：内核定义了类似 .initcall6.init 的特殊段，专门存放初始化函数指针。
  • 宏的作用：module_init(my_init) 展开后，会生成一个静态全局变量，其值为 my_init 函数地址，并通过 __attribute__((__section__(".initcall6.init"))) 强制放入指定段。
  • 链接脚本的作用：链接阶段，vmlinux.lds 脚本将所有目标文件中同名段（如 .initcall6.init 和 .initcall6s.init）的内容收集起来，首尾相接排列成一个连续数组，并用 __initcall6_start 和 __initcall6_end 标记首尾。
  • 启动时调用：do_initcalls() 遍历这个函数指针数组，依次调用每一个初始化函数。

💡 核心思想：利用编译时数据重排技术，将分散在无数文件中的初始化函数指针，汇聚成一个可遍历的数组，实现初始化的解耦与自动化调度。

二、Initcall 优先级全景图

除了 module_init，内核还提供了一系列优先级更高的初始化宏，以满足不同子系统之间的依赖顺序。

等级 宏定义 典型用途
early early_initcall(fn) 最早期的硬件初始化（SMP 前），如电源管理、DMA
0 pure_initcall(fn) 无任何依赖的“纯粹”初始化
1 core_initcall(fn) 核心子系统：调度器、内存管理
2 postcore_initcall(fn) 核心初始化后的补充初始化
3 arch_initcall(fn) CPU 架构相关初始化
4 subsys_initcall(fn) 内核子系统：I2C、SPI、PCI、USB 等
5 fs_initcall(fn) 文件系统初始化
(特殊) rootfs_initcall(fn) 根文件系统初始化
6 device_initcall(fn) 等价于 module_init，大多数设备驱动
7 late_initcall(fn) 晚期初始化，依赖所有子系统就绪（如 RTC）

⚠️ 关键认知：你的驱动之所以用 module_init，是因为它依赖的总线、GPIO 等基础设施已经由优先级更高的 subsys_initcall 或 arch_initcall 提前初始化完毕。

三、单片机裸机与 Linux 引脚配置区别

3.1 STM32 单片机（裸机/无操作系统）

  • 配置方式：引脚复用、GPIO 方向、时钟使能等全部硬编码在驱动源码内部。
  • 硬件耦合性：驱动代码与硬件接线强绑定。
  • 修改成本：一旦硬件引脚接线发生变化，必须同步修改驱动 C 代码并重新编译。

3.2 RK3568 Linux 平台（设备树 + 驱动分离）

  • 配置方式：所有引脚复用、GPIO 编号、时钟源等硬件描述信息统一存放在设备树（.dts/.dtsi）文件中。
  • 硬件耦合性：驱动代码不关心具体引脚编号，仅通过设备树接口（如 of_get_named_gpio）动态获取配置。
  • 修改成本：
    • 原厂板载硬件引脚配置已由 BSP 预设完成，移植常见外设驱动时无需修改驱动 C 代码。
    • 若实际硬件接线与默认设备树不符，唯一修改入口是设备树 DTS 文件，驱动 C 源码保持不动。

3.3 Linux 平台引脚修改准则

  • 驱动代码：与硬件无关，保持通用性和可移植性。
  • 设备树（DTS）：硬件描述的唯一事实来源。
  • 操作结论：引脚变更 → 仅改 DTS → 编译为 .dtb → 替换启动文件 → 驱动自动适配新引脚。

📌 今日总结：两个核心思维转变

  1. 从“函数调用”到“数组遍历” module_init 不是编译成一条 call 指令，而是生成一个函数指针存入特殊段，最终由内核统一遍历执行。这是内核实现启动流程自动化调度的关键设计。

  2. 从“代码配置”到“数据描述” 在裸机开发中，引脚是写死在代码里的常量；而在 Linux 驱动开发中，引脚变成了从外部数据文件读取的变量。这种 “硬件描述与驱动逻辑分离” 的设计，正是 Linux 能够支持数千种不同硬件板卡、而驱动代码依然保持简洁通用的核心哲学。
