OV-Watch 学习路径

前提

这份路线假设你是初学者——没摸过 STM32、没用过 FreeRTOS、也不知道 LVGL 是什么。没关系，顺着走就行。

原则：

  • 不用一次全看懂，每一遍比上一遍多懂一点就行
  • 能跑起来的代码 > 脑子里想象的概念
  • 改代码看效果 > 读代码猜逻辑

第一阶段：先跑起来（1-2 天）

这个阶段的目标只有一个：让仿真器在电脑上跑起来。暂时不要求理解代码。

1.1 什么是"仿真器"

lv_sim_vscode_win/ 这个目录的东西，可以让你在没有手表硬件的情况下，在 Windows 电脑上运行整个手表 UI。

原理很简单：

1
2

Keil 工程（真机）：   UI 代码 → LVGL → 调用 LCD 驱动 → 写到 ST7789 屏幕
仿真器工程（PC）：    同一份 UI 代码 → LVGL → 调用 SDL2  → 画到电脑窗口

UI 代码是完全一致的，区别只在于最底层画图的那个东西——一个是 LCD 驱动，一个是 SDL2。

1.4 第一个实验：改 HW_USE_HARDWARE

打开 lv_sim_vscode_win/user_test/Func/Inc/HWDataAccess.h，找到：

1

#define HW_USE_HARDWARE 0

这行代码的意思是：不访问真实硬件，全部用假的模拟数据。正因为设成了 0，仿真器才能在没有 STM32、没有传感器的情况下跑起来。

把它改成 1，重新编译，看会发生什么。

答案是：编译报错。因为 PC 上没有 STM32 的 HAL 库。这个实验让你直观理解硬件抽象层的意义——一层宏开关，隔开了 UI 代码和硬件代码。

第二阶段：搞懂 UI 层（2-3 天）

你已经在仿真器里看到了界面。现在搞清楚这些界面是怎么被创建、怎么被管理的。

2.1 LVGL 是什么

LVGL 是一个给单片机用的 GUI 库，跟 PC 上的 Qt、Web 前端本质上是一回事：

概念 LVGL 中叫什么 说明
屏幕 lv_scr_act() 当前显示的"画布"
按钮 lv_btn_create(parent) 创建一个按钮，放在 parent 上
文字 lv_label_create(parent) 创建一个文本标签
图片 lv_img_create(parent) 创建一个图片
刷新 lv_task_handler() LVGL 主循环，定期调用才会刷新画面

所有 LVGL 对象都是 C 语言通过函数调用创建的。没有 XML 布局文件，没有可视化编辑器（SquareLine Studio 只是辅助，最终导出还是 C 代码）。

2.2 拆解一个最简单的页面

建议从 ui_AboutPage.c 开始（关于页面，内容简单），用 VS Code 打开它。

不用逐行看懂，只需要回答三个问题：

  2. 这个页面有哪些屏幕对象？ → 看 lv_obj_t *ui_AboutPage 的声明
  3. 页面上的文字、图片在哪里创建的？ → 在 _screen_init() 函数体里找 lv_label_create / lv_img_create

2.4 LVGL 的"心跳"——lv_task_handler()

LVGL 不像网页有事件循环，它需要你每隔几毫秒主动调用一次 lv_task_handler()，才会更新画面。

在 user_TasksInit.c 里找到 LvHandlerTask，它每隔 1ms 调用一次 lv_task_handler()。这就是整个手表 UI 的心脏。

练习：改刷新频率

把 osDelay(1) 改成 osDelay(10)，重新编译运行。UI 会变卡顿。改回来。

这个实验让你理解——LVGL 的画面完全靠 lv_task_handler() 驱动，不调用它画面就静止。

第三阶段：理解 FreeRTOS 多任务（2-3 天）

第二阶段你看到了一个个任务（KeyTask、LvHandlerTask 等）。现在搞清楚"任务"到底是什么，以及这些任务怎么协作。

3.1 什么是 FreeRTOS 任务

单片机只有一颗 CPU，同一时刻只能做一件事。操作系统的作用就是让一颗 CPU 看起来像同时在做好几件事。

实现方式：每隔一段时间切换一次当前执行的任务（上下文切换），每个任务跑一小段就换成下一个。

核心概念

概念 代码中叫 一句话解释
任务 osThreadId_t 一个独立的执行流，有自己的 while(1) 循环
优先级 osPriority_t 谁先跑。高优先级的任务就绪时，低优先级的会被抢占
消息队列 osMessageQueueId_t 任务间传递数据的安全通道
延时 osDelay(ms) 当前任务暂停 ms 毫秒，让 CPU 去跑别的任务

3.2 逐个读懂 13 个任务

打开 user_TasksInit.c，对照下表找到每个任务：

高优先级组（不能卡，卡了就出大事）

任务 函数名 做什么 怎么验证
硬件初始化 HardwareInitTask 开机跑一次，把所有外设 init 完就挂起 看代码，任务末尾是 osThreadSuspend 还是 while(1)？
喂狗 WDOGFeedTask 每 100ms 喂一次独立看门狗 看门狗的作用：如果程序死循环了，看门狗收不到喂信号就复位 MCU
空闲检测 IdleEnterTask 用户多久没操作？超时就进睡眠 追踪 IdleTimerHandle 这个定时器
关机检测 StopEnterTask 长按 KEY1 后关机 找 power_shutdown 的调用点

中优先级组（用户交互相关）

任务 函数名 做什么
按键扫描 KeyTask 每 10ms 读一次 GPIO，判断哪个键被按下

低优先级组（可以慢，但不能错）

任务 函数名 做什么
LVGL 渲染 LvHandlerTask 循环调 lv_task_handler()，刷新画面
页面切换 ScrRenewTask 接收按键消息，管理页面栈的 push/pop
传感器通用 SensorDataTask 定时读温湿度、气压、罗盘
心率采集 HRDataTask 读 EM7028 + 跑心率算法
充电检测 ChargPageEnterTask 检测到充电就弹充电动画
蓝牙消息 MessageSendTask 通过蓝牙把数据发出去
姿态检测 MPUCheckTask 抬腕检测，决定要不要唤醒屏幕
数据保存 DataSaveTask 改设置后，写到 EEPROM 持久化

核心理解

为什么 LVGL 渲染优先级最低？ 因为渲染耗时长。如果优先级高，按键和传感器数据就来不及处理，体验会变差。宁可画面慢一帧，也不能按键没反应。

3.3 任务间怎么通信：消息队列

任务之间不能直接调用对方的函数（不安全），而是通过消息队列传递数据。

// 发送方

// 接收方
osMessageQueueGet(Key_MessageQueue, &key_value, NULL, osWaitForever);

队列名 谁发 谁收 传递什么
Key_MessageQueue KeyTask ScrRenewTask 哪个键被按下
Idle_MessageQueue 定时器 IdleEnterTask 无操作计时

4.1 硬件抽象层（HWDataAccess）

文件：HWDataAccess.h

问题：为什么需要这一层？

想象一下，如果没有这一层——每个 UI 页面都直接调用 HAL_I2C_Read(MPU6050_ADDR)。那么在 PC 仿真器上：

  • 没有 I2C 总线 → 编译不过
  • 没有 HAL 库 → 链接不过
  • 有了也读不到数据 → 运行崩溃

硬件抽象层的解决方案很巧妙：

1
2

#define HW_USE_HARDWARE 1   // 模式 1：编译真实驱动代码
#define HW_USE_HARDWARE 0   // 模式 0：返回写死的模拟数据

它是怎么实现的

打开 HWDataAccess.h，看这段：

  #define HW_USE_AHT21  1    // 需要真实温湿度驱动
  // ...
#endif

为什么要存函数指针而不是直接存页面对象

4.3 双栈计算器（StrCalculate）

文件：StrCalculate.c

算法核心

两个栈：操作数栈（存数字）和运算符栈（存 +-*/）。

遍历输入字符串，每读到一个字符：

1
2
3
4
5

读到数字 → 压入操作数栈
读到运算符 → 跟运算符栈顶比较优先级：
  - 当前 > 栈顶 → 入栈（等着）
  - 当前 ≤ 栈顶 → 弹栈计算（栈顶的优先算）
读完所有字符 → 弹空运算符栈，结果在操作数栈顶

为什么要建两个栈不直接算

因为运算符有优先级。比如 1+2*3：

  • 读到 + 时还不知道后面有 *，不能直接算 1+2
  • 需要把 + 存起来，让 * 先算

栈就是解决"后面的事比前面急"的数据结构。

小数怎么处理

代码里用 NumBehindPoint_Flag 标记当前读到小数点后第几位。核心逻辑：

2
3

// 小数点后第一位：÷10
// 小数点后第二位：再÷10（相当于 ÷100）
// 然后加上栈顶已有的整数部分

4.4 心率算法

文件：HrAlgorythm.c

PPG 信号是什么

EM7028 传感器发射绿光穿过皮肤，光电二极管接收反射光。心脏跳动时血液量变化 → 反射光强度变化 → 产生一个波形。

峰值检测算法的思路 

1
2
3
4

原始信号采样 → 环形队列缓冲（存最近 N 个点）
  → 扫描队列找峰值（比前后都高）
  → 计算相邻两个峰间距（多少个采样点）
  → 换算为 BPM（两次心跳的间隔 → 每分钟跳多少次）

比官方库快的原因是：官方库可能做了 FFT 等复杂运算，而峰值检测只需遍历一次队列。

环形队列

文件：user_Queue.c

环形队列是一个固定大小的数组，写满了就覆盖最老的数据。适合传感器这种"一直来新数据，只关心最近 N 个点"的场景。

5.1 为什么要分 BootLoader 和 APP

没有 BootLoader 时，更新固件必须用 ST-Link 物理连接到板子。有了 BootLoader，可以通过蓝牙无线升级。

二者的关系：

1
2
3
4
5

上电 → BootLoader（先跑）→ 检测：要升级吗？
  ├─ 要 → 接收新固件 → 写入 FLASH
  └─ 不要 → 直接跳转到 APP

APP（主程序）正常运行

5.2 FLASH 怎么分的

STM32F411 有 512KB FLASH：

1
2
3
4
5
6
7

0x08000000 ┌──────────────┐
           │  BootLoader  │ 48KB（先跑，管升级）
0x0800C000 ├──────────────┤
           │              │
           │    APP       │ 464KB（主程序）
           │              │
0x0807FFFF └──────────────┘

为什么从 0x0000C000 开始

0xC000 = 48 × 1024 = 49152 字节。BootLoader 占前 48KB，APP 从第 49KB 开始。

5.3 中断向量表重映射

BootLoader 跑的时候，CPU 的中断向量表指向 0x08000000。APP 跑的时候需要指向 0x0800C000。

代码：main.c:75

1

SCB->VTOR = 0x0000C000U;

这是 APP 的 main() 函数里必须第一行执行的语句。没有它，中断一来 CPU 还去 BootLoader 的向量表取地址，后果不可预期。

5.4 Ymodem 协议

文件：ymodem.c

Ymodem 是一个文件传输协议。发送方把文件切成 128 字节的小包，每包加 CRC 校验。接收方校验通过就 ACK，校验失败就 NAK 要求重发。

核心流程 

1
2
3
4
5
6
7
8
9

发送方（手机）                  接收方（手表 BootLoader）
    │                                │
    │──────── 第 1 包(128B+CRC) ────→│ 校验 ✓
    │←────────── ACK ───────────────│
    │──────── 第 2 包(128B+CRC) ────→│ 校验 ✓
    │←────────── ACK ───────────────│
    │             ...                │
    │──────── 最后包(EOT) ──────────→│
    │←────────── ACK ───────────────│  写入完成，跳转 APP

  3. 把 OTA 流程用自己的话写一遍（不需要理解 Ymodem 每行代码，能讲清"上电→检测→接收→写 FLASH→跳转"就行）

怎么知道驱动写得对不对

打开 BSP/MPU6050/MPU6050.c（举例）：

  1. 找 HAL_I2C_Mem_Read() 或 HAL_I2C_Mem_Write() 调用
  2. 看第一个参数是 I2C 句柄，第二个参数是设备地址
  3. 地址必须跟原理图上的一致

练习
