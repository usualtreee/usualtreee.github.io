一、核心概念扫盲

1. .h / .so / .cpp 的关系 

1
2
3

.h（头文件）     →  函数声明，告诉编译器"有这个函数"
.cpp/.cc（源文件）→  函数实现，具体代码
.so（动态库）    →  .cpp 提前编译好的二进制，运行时加载
.h .cpp .so
谁写 库作者 库作者 库作者编译产出
你需要干嘛 #include 进来 跟项目一起编译，或拿 .so 代替 运行时加载
编译时需要 是 是（或 .so） 链接时需要

一句话：.h 是菜单，.so 是后厨。你拿着菜单点菜（编译），后厨负责做（运行），缺一个都吃不上饭。

2. 编译 vs 链接 

1
2

编译：.cpp → .o（目标文件）
链接：.o + .so → 可执行文件

3. 静态库 vs 动态库

静态库 .a 动态库 .so
链接时机 编译时塞进可执行文件 运行时临时加载
可执行文件大小 大 小
升级库 要重新编译 替换 .so 就行

4. .h vs .hpp / .cpp vs .cc

  • .hpp = 纯 C++ 头文件（有 class/template），.h = C/C++ 通用
  • .cc = .cpp，同一个东西，Google 代码规范喜欢用 .cc

5. 编译命令参数 

1

g++ main.cpp -I/opt/rknn/include -L/opt/rknn/lib -lrknn_api -o test
参数 含义
-I 头文件搜索路径（找 .h）
-L 库搜索路径（找 .so 的目录）
-l 链接哪个库（找 lib + 名字 + .so）
-o 输出文件名

二、两套 MPP 库冲突问题

SONAME 机制

每个 .so 内部有 SONAME 字段：

1
2
3
4
5
6

编译时：链接器读到 SONAME = librockchip_mpp.so.1
        → 写进可执行文件："我需要 librockchip_mpp.so.1"

运行时：系统按 SONAME 找文件
        → 找不到 .so.1 → 退到 /usr/lib/ 找
        → 找到系统旧版 → 版本不匹配 → 段错误

你的板子有两套 

1
2

third_party/mpp/lib/librockchip_mpp.so  ← SDK 新版（跟项目匹配）
/usr/lib/.../librockchip_mpp.so.1       ← 系统旧版（刷固件时自带的）

程序优先找 .so.1，在 third_party 里找不到，落到系统路径 → 加载旧版 → 崩。

解决 

1
2
3

cd third_party/mpp/lib
ln -sf librockchip_mpp.so librockchip_mpp.so.1
ln -sf librockchip_mpp.so librockchip_mpp.so.0

创建软链接，让程序找 .so.1 时自动指向正确的 .so。

三、RK3588 NPU 工作机制

NPU 需要 rknn_server 后台服务 

1

程序 → librknnrt.so → rknn_server（常驻进程）→ /dev/rknpu → NPU 硬件

rknn_server 是用户态后台服务，负责统一调度多程序对 NPU 的访问。不启动 = 段错误。

1
2

sudo cp third_party/rknn/bin/rknn_server /usr/bin/
sudo rknn_server &

RK3588 NPU 架构

  • 3 个 NPU 核心（Core 0/1/2），每个 2 TOPs
  • 本项目：人员检测 → Core 0，安全帽 → Core 1，打电话 → Core 2
  • 同一帧 3 个模型真正并行（3 核同时算）
  • 4 路流同类模型排队串行（时分复用同一个核）

多核绑定方式 

1

rknn_set_core_mask(rkModel, RKNN_NPU_CORE_0);  // 绑到 Core 0

四、RGA 硬件加速与 fallback

RGA 为什么需要物理地址

CPU 用虚拟地址，MMU 自动翻译。RGA 是硬件模块，不经过 MMU，必须拿物理地址。

物理地址 = 0 → 分配 DMA buffer 失败 → Invalid argument。

问题原因

SDK 自带的 RGA 库跟板子内核驱动 ioctl 接口版本不匹配。

解决 

1
2
3
4
5
6

# 换系统自带的 RGA 库（跟内核一起编译的，版本匹配）
mv third_party/rga/lib/librga.so third_party/rga/lib/librga.so.bak
cp /usr/lib/aarch64-linux-gnu/librga.so.2.1.0 third_party/rga/lib/librga.so

# 放权 DMA heap 设备
chmod 666 /dev/dma_heap/*

为什么不换也能跑

代码里有 fallback：

1
2
3
4

if (!rga_done) {
    cv::cvtColor(ori_img, img, cv::COLOR_BGR2RGB);  // CPU 替代
    cv::resize(img, img, cv::Size(width, height));
}

RGA 是加速手段，不是必须路径。

二、C++多线程三件套

【std::thread】

1
2
3

std::thread t1(func);   // 创建对象，传入函数
t1.join();              // 主线程等子线程结束
t1.detach();            // 子线程独立运行（守护线程）

【std::mutex + std::lock_guard】

  • std::mutex 是类（互斥锁本体）
  • std::lock_guard 是类模板实例化出的对象
  • RAII：构造时lock()，析构时自动unlock()，防止死锁

【std::condition_variable】

  • 类，对象调用wait()/notify_one()
  • 必须用std::unique_lock（支持中途解锁），不能用lock_guard
  • wait(lock, lambda)：lambda返回true才真正醒来，防虚假唤醒
  • 有界阻塞队列 = mutex + condition_variable(not_full/not_empty) + while循环

【Lambda [this]】

  • 匿名函数，[this]捕获列表表示"把当前对象抓进来，能用成员变量"
  • 传给wait()作为"醒来条件判断函数"

四、视频编解码常识

  • H.264/H.265：视频压缩标准；H.265同画质省50%带宽，解码更累
  • 解码 = 压缩数据→原始像素（YUV）；编码 = 原始像素→压缩数据
  • I帧：完整图像，不依赖任何帧
  • P帧：依赖前面帧；B帧：依赖前后帧
  • 解码必须从I帧开始，否则残差帧无参考基准会花屏
  • 直播推流通常关闭B帧（增加延迟），GOP设15~30

五、嵌入式/AI核心八股

【DMA vs MMU】

DMA MMU
干啥的 硬件搬运数据（不经CPU） CPU虚拟地址→物理地址翻译
为谁服务 外设（RGA/NPU/解码器） CPU
地址 只认物理地址 处理虚拟地址
RK3588 两者都有 两者都有
STM32F411 有DMA 无MMU，只有MPU

Cache一致性：CPU写Cache后DMA读DDR可能读到旧值 → 用dma_alloc_coherent()分配非缓存内存，或手动sync

【PTQ（训练后量化）】

  • 只做一次：FP32模型→校准集统计→生成INT8 .rknn文件
  • 推理时每帧实时查预存Scale/Zero-Point，PTQ本身不重复做
  • 不能再二次量化INT8→INT4；精度不够时回退FP16或混合精度
  • 对称量化（ZP=0，适合权重）vs 非对称量化（ZP偏移，适合ReLU激活值）
  • Per-Tensor（整个层一个Scale）vs Per-Channel（每通道独立Scale，精度更高）

【RGA】

  • 硬件版OpenCV imgproc：只做resize、旋转、格式转换、裁剪
  • 不是完整OpenCV，不支持滤波/神经网络/绘图/读文件
  • 调用方式：填结构体(rga_info_t)→发命令→硬件异步执行→CPU零占用
  • 必须用dmabuf fd + 物理连续内存，不能用cv::Mat虚拟地址

【NMS / 多模型融合】

  • NMS：同一模型内去重，IoU>阈值只留置信度最高的框
  • IoU = 交集/并集；IoM = 交集/最小框面积（处理人帽嵌套）
  • 多模型融合：跨模型IoU去重 + IoM处理嵌套 + 加权置信度融合（按mAP分配权重）

六、为什么生产部署用C++而不是Python？

  • RKNN/MPP/RGA板端API只有C/C++，Python绑定性能差、延迟高
  • 视频硬解码（MPP）无Python接口，必须C/C++
  • Python GIL假并行，实时视频帧率上不去
  • C++编译后单二进制，启动快、内存小
  • Python仅用于：PC端训练（PyTorch）、模型转换（RKNN-Toolkit2）、精度验证
