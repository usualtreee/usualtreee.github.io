全志桌面屏幕 — 学习路径

前提

  • Windows 电脑一台（你现在就有的）
  • 不需要买开发板就能开始学
  • 目标不是"学完所有理论"，是"能看懂/能改/能加功能"

第一阶段：让代码在 WSL 里跑起来（1-2 天）

目标：不买硬件，在 Windows 上看到这个项目的 GUI 窗口。

学习内容：

  1. 装 WSL2（Windows Subsystem for Linux）
    • PowerShell 管理员运行：wsl --install -d Ubuntu-22.04
    • 重启电脑，打开 Ubuntu 终端
  2. 装依赖： 
    1
2

    sudo apt update
sudo apt install build-essential cmake libsdl2-dev
  3. 编译运行： 
    1
2
3

    cd /mnt/c/Users/13403/Desktop/quanzhi-LVGL/app_sdk_v2/app_sdk
./build.sh -linux
./build/app/demo
  4. 如果跳出 SDL 窗口显示 GUI 界面 → 成功了

获取的技能：知道程序是怎么从源码变成可运行的窗口的。

不需要理解：CMake、编译原理、交叉编译——先跑起来再说。

第二阶段：看懂 main.c 和 LVGL 怎么转起来的（1-2 天）

目标：看懂程序入口，能加一个自己的 UI 页面。

学习内容：

  1. 读 app/main.c（45 行）

    • lv_init() → 初始化 LVGL
    • lv_port_disp_init() → 初始化显示
    • while(1) { lv_task_handler(); usleep(...); } → 主循环（死循环刷新 UI）
    • 不需要看懂 WiFi/BLE 的初始化，先跳过

  2. 读 app/ui/page_about_us.c（97 行，项目里最短的页面）

    • com_style_init() → 背景样式
    • lv_obj_create(parent) → 创建容器
    • lv_img_create() → 创建图片
    • lv_obj_set_align() → 对齐
    • 模式：创建容器 → 往里面放控件 → 对齐 → 完成

  3. 改点东西练手

    • 把关于页的文字改掉
    • 在主页菜单里加一项"测试"，点了跳转到关于页
    • 重编译看效果

获取的技能：看懂 LVGL 的页面是怎么建的，能改文字、加按钮。

关键概念：

  • lv_obj_t * → LVGL 里所有东西（按钮、文字、图片）都是"对象"，返回指针
  • lv_obj_set_align(obj, LV_ALIGN_CENTER, x, y) → 把对象放在父容器的某个位置
  • delete_current_page() → 删掉当前页面的所有控件，清屏

第三阶段：看懂一个完整的业务页面（2-3 天）

目标：理解 UI 页面 + 数据 + 逻辑是怎么配合的。

学习内容：

  1. 读 app/ui/page_tomato_learn.c（194 行，番茄钟学习页面）

    • 页面创建：init_page_tomato_learn() → 建容器、标签、按钮
    • 按钮回调：按钮点击后调 btn_event_cb()，里面修改 device_state_t 里的状态
    • 定时器刷新：每 1000ms 更新倒计时文字，归零时跳转页面
    • 音频：开始学习时播 audio_start.wav

  2. 读 app/ui/data/device_data.h

    • 全局只有一个 device_state_t 结构体
    • 所有页面读写的数据都在里面
    • get_device_state() 获取，改完自动生效

  3. 读 app/ui/page_conf.h + page_conf.c

    • 每个页面注册为 init_page_xxx() 函数
    • 页面跳转就是调另一个 init_page_xxx()

获取的技能：能独立加一个新页面，带按钮和跳转。

关键概念：

  • 全局状态：所有页面共享 device_state_t，不需要传参
  • 页面跳转：清掉当前页 → 调目标页的 init 函数 → 新页面显示
  • 定时器：lv_timer_create(callback, interval, user_data) 创建周期性任务

第四阶段：看懂异步通信模式（1-2 天）

目标：理解三个异步模块怎么工作的。

学习内容：

  1. 读 component/osal/osal_thread.c + osal_queue.c（约 250 行）

    • 就是对 pthread 和 System V 消息队列的简单包装
    • osal_thread_create() → 开一个线程
    • osal_queue_send() → 发消息到队列
    • osal_queue_recv() → 从队列收消息

  2. 读 component/player/audio_player_async.c（84 行，最短的异步模块）

    UI 页面调 start_play_audio_async("xxx.wav")

  3. 对比 component/net/http_manager.c 和 component/usb_hid/usb_hid_async.c

    • 三个模块结构完全一样，只是工作线程干的事不同
    • 看懂一个 = 看懂全部

获取的技能：理解项目中反复出现的"异步模式"。

关键概念：

  • 同步 vs 异步：同步 = 调完等结果（UI 会卡），异步 = 发了不管自己继续跑
  • 消息队列：一个"邮箱"，一方投递一方取件，互不阻塞

第五阶段：理解跨平台是怎么回事（1 天）

目标：知道为什么同一份代码能在 PC 和板子上跑。

学习内容：

  1. 看 app/main.c 里的 #ifdef SIMULATOR_LINUX

    • 编译时如果定义了 SIMULATOR_LINUX → 跳过 WiFi 和 BLE
    • 没定义 → 启动 WiFi 和 BLE

  2. 看 component/hal/ 目录下的 6 个头文件

    • 只定义接口，不写实现
    • 比如 em_play_audio() 的声明在这里，实现在平台层

  3. 对比两个平台的音频实现：

    • platform/x86linux/src/hal/em_hal_audio.c → system("aplay xxx.wav &")
    • platform/t113/src/hal/em_hal_audio.c → fork() + system("aplay") + GPIO 控制功放
    • 接口一样，实现不同

获取的技能：理解 HAL（硬件抽象层）的套路——定义统一接口，不同平台不同实现。

关键概念：

  • 条件编译：#ifdef 决定哪段代码参与编译
  • HAL 模式：上层调接口，底层换实现，代码不改

第六阶段：摸到真机（装 WSL + 板子到手后）

目标：把代码烧到 T113 板子上跑。

需要的工具：

  • T113 板子一块
  • USB 数据线
  • PC 上装 ADB（sudo apt install adb）

步骤：

  1. ./build.sh -t113 → 交叉编译出 ARM 版的 demo
  2. adb push build/app/demo /usr/bin/ → 推到板子上
  3. adb push build/app/res/* /usr/res/ → 推资源文件
  4. adb shell /usr/bin/demo & → 在板子上跑

获取的技能：知道交叉编译和推送部署的流程。

学习原则

  1. 先抄后改再创：先照抄一个页面，改文字，加按钮，最后才自己从零写
  2. 不需要全懂再动手：看到一个 API 就去 LVGL 文档 搜，用到了再学
  3. 跑起来比看懂重要：改完立刻编译运行看效果，有反馈才有动力
  4. 跳过不用的：BLE 协议细节、cURL 内部实现、System V IPC 原理——改 UI 用不到就不看
  5. 一份代码拆成两层看：
    • 上层（app/、component/）：业务逻辑，跨平台通用
    • 下层（platform/）：硬件相关，换平台才改
