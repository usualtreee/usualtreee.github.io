第一部分：项目概况

项目是什么

基于全志 T113（ARM Cortex-A7）平台，从零独立开发智能桌面助手终端固件。功能如下：

HAL 硬件初始化 → WiFi/BLE 通信子系统启动 → HTTP 异步天气获取 → LVGL 多页面 GUI 渲染 → 触屏/按键交互

具体功能：

  • 多表盘时钟主页（数字/大字/模拟表盘 3 种样式，天气显示，NTP 校时）
  • 闹钟提醒（倒计时 + 定时闹钟，弹窗 + WAV 音频播报）
  • 番茄钟学习计时器（学习/休息双阶段，暂停恢复，完成播报）
  • 场景联动（BLE 控制 5 类 IoT 子设备：RGB 灯、开关、人体传感器、杯垫、火焰传感器）
  • 语言学习游戏（14 种声音图标，点击播报中文发音）
  • 天气查询（心知天气 API，18 城市可选）
  • WiFi 配置（wpa_supplicant 管理，SSID/密码输入，连接状态反馈）
  • USB HID 键盘模拟（Ctrl+Alt 组合键，10 个快捷应用一键启动）
  • 系统设置（亮度/音量调节、横竖屏切换、重启）
  • PC 仿真调试（SDL2 + CMake），无需真实硬件即可开发 UI

核心数据通路说明

  1. BLE 通信链路：BLE 模组 UART ↔ ble_mesh.c（AT 指令 + 自定义协议）→ 更新 device_state_t → UI 页面刷新设备状态/电量
  2. WiFi 管理链路：wpa_manager 通过 wpa_ctrl socket 控制 wpa_supplicant → 状态回调 → 更新 device_state_t.wifi_* → UI 状态图标刷新；连接成功后自动触发 NTP 校时 + 天气获取
  3. HTTP 天气获取：http_manager 异步线程 → cURL GET 心知天气 API → cJSON 解析 → 回调更新 device_state_t.weather_* → 主页天气文字刷新
  4. UI 渲染：主循环 lv_task_handler() + usleep 按需调用 → LVGL 计算脏区域 → sunxifb 刷新 framebuffer → G2D 硬件加速混合
  5. 页面导航：调用目标页面的 init_page_xxx() → 内部先调 delete_current_page() 清理旧页面 → 在新 lv_scr_act() 上创建控件；弹窗类页面通过 lv_layer_top() 叠加显示
  6. 音频播放：UI 页面调 start_play_audio_async() → OSAL 消息队列 → 工作线程 → HAL em_play_audio() → T113 上 fork 子进程执行 aplay + GPIO34 控制功放
  7. 参数持久化：device_param_write() → 将整个 device_state_t 通过 file_param_write() 原子写入 param.cfg（先写 temp 文件再 rename）

第二部分：关键设计要点

1. 硬件抽象层（HAL）

component/hal/ 目录下 6 个头文件定义了所有硬件操作的统一接口。通过 CMake 条件编译 #ifdef SIMULATOR_LINUX 在平台层切换实现：

平台 音频 亮度 系统 时间 串口 控制台
T113 amixer + aplay + GPIO34 sysfs backlight CPU信息/重启 NTP + settimeofday termios PTS
x86 Linux amixer + aplay 后台 空实现 空实现 NTP + settimeofday termios 空实现

UI 层和组件层代码完全不接触硬件寄存器或 sysfs，实现了"写一次 UI，双平台运行"。

2. 统一异步组件模式

Audio Player、HTTP Manager、USB HID 三个模块采用完全相同的异步架构：

1

[调用方] → osal_queue_send(cmd) → [System V 消息队列(容量50)] → osal_queue_recv() → [工作线程] → HAL 接口

以 Audio Player 为例：

  • audio_obj 结构体：{AUDIO_COMM_ID id, char file_name[256]}
  • start_play_audio_async(url) — 调用方（任何 UI 页面）发消息到队列，立即返回不阻塞
  • 工作线程 audio_thread_fun — 循环收消息 → em_stop_play_audio() → em_play_audio(file_name)

三个模块的区别仅在于：

模块 命令类型 HAL 调用
audio_player START / STOP em_play_audio() / em_stop_play_audio()
http_manager GET_WEATHER / GET_TIME cURL GET + cJSON 解析
usb_hid CTRL_ALT / CTRL HID report 写入 /dev/hidg0

线程数少——每个模块只开 1 个工作线程，不使用线程池，简洁够用。

3. BLE 模组通信与自定义协议

物理层

BLE 透传模组通过 UART 与 T113 通信：

  • T113 真机：/dev/ttyS0，115200 baud
  • 仿真器：/dev/ttyUSB0

AT 指令层

模组通过 AT 指令集控制：设置设备名（AT+NAME）、设置 MAC（AT+MAC）、自动重连（AT+AUTO_CFG）、扫描自定义广播（AT+SCAN_MANU）等。

自定义二进制协议（透传通道）

控制指令和状态上报使用统一帧格式：

1

0xAA [包长度] [设备类型] [数据...] 0x55

5 种设备类型：

ID 类型 方向 控制帧 上报帧
1 RGB灯+开关 可控 AA 06 01 R G B state 55 AA 07 01 R G B state battery 55
2 舵机 可控（已合并到类型1） — —
3 火焰传感器 仅上报 — AA 04 03 status battery 55
4 人体传感器 仅上报 — AA 04 04 status battery 55
5 智能杯垫 仅上报 — AA 04 05 status battery 55

软件架构

双线程 + System V 消息队列：

1

UART 读线程 → SysV msg queue (key=123) → 协议处理线程 → device_state_t → UI 刷新

4. 全局设备状态管理

所有设备状态集中在 device_state_t 结构体中（device_data.h），包含：

  • 5 设备连接状态 + 电量
  • RGB 颜色值 + 开关状态
  • 传感器数据 + 告警配置（阈值/间隔）
  • 番茄钟时间配置 + 剩余时间
  • 天气信息 + 城市
  • WiFi 连接状态
  • 系统设置（亮度/音量/横竖屏/表盘样式/闹钟）
  • 版本号

状态通过 get_device_state() 全局访问，通过 device_param_write() 持久化到 param.cfg。

原子化持久化

file_param_write() 采用"先写 temp 再 rename"策略防止写入中断导致配置文件损坏：

5. 页面导航与弹窗叠加

普通页面导航

导航通过函数调用实现，无显式页面栈：

1
2

delete_current_page(style);  // 清理当前屏幕所有控件

弹窗/叠加层

闹钟提醒、告警、WiFi 加载、屏保等页面使用 lv_layer_top() 创建，覆盖在所有正常页面之上：

1

lv_obj_t *scr = lv_obj_create(lv_layer_top());  // 顶层叠加

弹窗通过单例守卫防止重复创建：

1
2
3

static lv_obj_t *page = NULL;
if (page != NULL) return;   // 已存在，不重复创建
page = lv_obj_create(lv_layer_top());

第三部分：踩坑记录与解决方案

踩坑 1：BLE 模组 UART 通信丢包与粘包

现象

UART 接收 BLE 模组数据时，出现半包、粘包，导致协议解析错误，设备状态无法更新。

根因

BLE 模组通过 UART 输出 AT 指令响应 + 透传数据，两种数据混杂在同一串口。UART 读线程每次只读一段，可能读到不完整的帧。

解决

使用 System V 消息队列解耦：UART 读线程只管读原始数据并发送到消息队列，协议处理线程从队列取出数据，通过帧头 0xAA / 帧尾 0x55 边界检测进行拼包和拆包，确保每次处理的都是完整帧。

踩坑 2：WiFi 密码错误无 UI 反馈

现象

输入错误密码后，WiFi 加载页面一直转圈，用户不知道发生了什么。

根因

wpa_supplicant 在密码错误时发出 CTRL-EVENT-SSID-TEMP-DISABLED 事件，但初始版本的事件处理未解析此消息。

解决

在 wpa_manager 事件线程中增加对该事件的解析，映射到 WPA_WIFI_WRONG_KEY 状态，回调 UI 显示"密码错误"提示。

踩坑 3：LVGL 模拟器窗口坐标偏移

现象

SDL2 模拟器窗口中，鼠标点击位置与实际 UI 控件位置有偏移，无法正常交互。

根因

SDL2 窗口大小与 LVGL 配置的 MONITOR_HOR_RES / MONITOR_VER_RES 不匹配，输入驱动未做坐标缩放。

解决

在 monitor.c 的鼠标读取函数中添加 zoom 系数：lv_x = SDL_x * LVGL_HOR_RES / SDL_window_width。代码中的 SDL_MAIN_HANDLED 宏定义表明作者在 Windows 上遇到过 WinMain 链接问题并已修复。

踩坑 4：闹钟弹窗重复创建导致界面混乱

现象

闹钟触发时，弹窗偶尔出现多个叠加，关闭一个后底下还有。

根因

闹钟检测在定时器回调中执行，1 秒一次。弹窗未关闭期间定时器仍在触发，导致重复创建。

解决

弹窗函数加单例守卫——静态指针 static lv_obj_t *page = NULL，创建前置检查 if (page != NULL) return，关闭时置 NULL。告警弹窗、屏保页同样采用此模式。

踩坑 5：配置文件断电损坏

现象

设备异常断电后重新上电，所有设置恢复默认值，之前修改的亮度、音量、闹钟等全部丢失。

根因

直接 fwrite 到 param.cfg，写入过程中断电会导致文件内容不完整或损坏。

解决

改用"先写临时文件再 rename"的原子化写入：数据写入 param.cfg_copy → fsync 确保落盘 → rename 原子替换为目标文件。读取时增加数据有效性校验。

踩坑 6：T113 音频播放多次触发导致进程僵尸

现象

频繁触发音频播放后，系统出现僵尸进程（aplay defunct），最终无法播放。

根因

T113 上 em_play_audio() 通过 fork() 子进程执行 aplay，父进程未 waitpid() 回收子进程。多次播放产生多个僵尸。

解决

播放前先检查 aplay_pid，如果已有进程在播放则 kill + waitpid 回收后再 fork 新进程：
