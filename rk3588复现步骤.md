# 第一部分：项目概况

## 项目是什么

这是一个基于 **瑞芯微 RK3588** 的**多路视频 AI 智能分析系统**。功能是：

> 4 路视频流 → 硬件解码 → NPU 并行跑 3 个 AI 模型检测 → 画面拼接 → 编码推流 → 远程观看

具体检测内容：

- 人员检测（person）
- 安全帽佩戴检测（helmet）
- 打电话行为检测（callplay）

4 路画面拼接成 2×2 的合成画面，通过 RTMP/RTSP/GB28181 推流到外部平台，用 VLC 或网页实时观看。

## 技术栈

| 组件 | 用途 |
|------|------|
| Rockchip MPP | 芯片内置硬件视频解码 + 编码 |
| RKNN (NPU) | 芯片内置 6TOPS AI 推理单元 |
| RGA | 芯片内置硬件图像缩放/格式转换 |
| OpenCV | 软件图像处理、绘制检测框 |
| FFmpeg | 视频文件解封装、RTMP/RTSP 推流 |
| eXosip2/osip2 | GB28181 国标 SIP 协议栈 |
| C++17 / CMake | 构建系统 |

# 第三部分：踩坑记录与解决方案

实际部署中共遇到 **4 个关键问题**，以下是完整排查过程。

## 踩坑 1：NPU 推理段错误（rknn_server 未启动）

### 现象

```bash
$ ./build/myDemo 1 ./assets/videos/1.mp4
StreamLoaderManager created
段错误 (核心已转储)
```

### 排查过程

```bash
# 查 NPU 驱动
lsmod | grep rknpu        # 无输出（驱动是编进内核的，不是模块）

# 查 NPU 设备节点
ls -la /dev/rknpu*        # 不存在！

# 查内核日志
dmesg | grep -i npu
# 输出：RKNPU fdab0000.npu: ...  ← 内核驱动已初始化
# 但设备节点没创建 → 用户空间 rknn_server 没跑
```

### 根因

RKNN Runtime 的架构不是程序直接操作 NPU 硬件，中间隔了一层：

```
程序 → librknnrt.so → rknn_server（常驻进程）→ /dev/rknpu → NPU 硬件
```

`rknn_server` 是用户空间的后台服务进程，负责管理 NPU 资源队列。如果没有它，`librknnrt.so` 找不到服务端，后续任何 NPU 调用都是空指针 → 段错误。

### 解决

```bash
# 从项目 third_party 里找到 rknn_server
find /opt/demo_multhread_decode_infer_mulmodel -name "rknn_server"
# → third_party/rknn/bin/rknn_server

# 拷贝到系统路径并启动
sudo cp third_party/rknn/bin/rknn_server /usr/bin/
sudo rknn_server &
ps aux | grep rknn_server    # 确认已启动
```

## 踩坑 2：MPP 硬件解码段错误（库版本不匹配）

### 现象

rknn_server 启动后，还是段错误。用 GDB 定位：

```bash
gdb --args ./build/myDemo 1 ./assets/videos/1.mp4
(gdb) run
Program received signal SIGSEGV
#0  h264e_vepu_stream_amend_deinit (ctx=0x0) at hal_h264e_stream_amend.c:94
(gdb) bt
#1  hal_h264e_vepu580_deinit
#2  hal_h264e_vepu580_init
#6  mpp_init               ← MPP 硬件编码器初始化崩溃
#7  MppEncoder::Init       ← 编码器创建时传入空指针
```

### 排查过程

```bash
# 程序编译时 RPATH 已指向 third_party/mpp/lib
# 但 ldd 显示加载了系统版本
ldd ./build/myDemo | grep mpp
# → librockchip_mpp.so.1 => /usr/lib/aarch64-linux-gnu/librockchip_mpp.so.1

# 原因：third_party/mpp/lib/ 里只有 librockchip_mpp.so
# 没有 .so.1 版本后缀！
ls third_party/mpp/lib/
# → librockchip_mpp.so  librockchip_mpp.a  libmpi_enc.so
#     ^^^^^^^^^^^^^^^^^^ 没有 librockchip_mpp.so.1
```

### 根因

程序编译时链接的是 `librockchip_mpp.so.1`，动态链接器在 `third_party/mpp/lib/` 里找不到这个名字，就跳过该路径，去 `/usr/lib/` 找了系统自带的旧版 MPP 库。两套 API 不兼容 → 空指针 → 段错误。

### 解决

```bash
# 补上版本化的软链接
cd third_party/mpp/lib
sudo ln -sf librockchip_mpp.so librockchip_mpp.so.1
sudo ln -sf librockchip_mpp.so librockchip_mpp.so.0

# 验证现在加载的是 third_party 版本
ldd build/myDemo | grep mpp
# → librockchip_mpp.so.1 => .../third_party/mpp/lib/librockchip_mpp.so.1  ✓
```

## 踩坑 3：普通用户 vs root 权限问题

### 现象

rknn_server 以 root 启动，普通用户跑程序可能无法与它通信。直接用 `./build/myDemo` 偶尔还是段错误。

### 解决

用 root 跑程序（rknn_server 是 root 启动的，内核设备节点也是 root 权限）：

```bash
sudo su -c 'MYDEMO_RTMP_URL="..." ./build/myDemo 1 ./assets/videos/1.mp4'
```

## 踩坑 4：RGA 硬件缩放报 Invalid argument（不致命，影响性能）

### 现象

程序跑起来了，但频繁打印：

```
RgaBlit(1513) RGA_BLIT fail: Invalid argument
rga_task_submit(2171): Failed to call RockChipRga interface
buffer[handle,fd,va,pa] = [0, 0, 0x..., 0]   ← pa（物理地址）全是 0
```

### 根因

RGA（硬件图像缩放）需要 DMA 物理连续内存。所有 buffer 的物理地址（pa）都是 `0`，说明 DMA buffer 没有成功分配到物理地址。3 种可能：

1. `third_party/rga/lib/librga.so` 版本跟板子内核 RGA 驱动不匹配 → ioctl 参数对不上
2. DMA heap 设备权限问题（`/dev/dma_heap/*` 默认只 root 可访问）
3. CMA 内存不足（程序需要约 128MB，实际可用 127MB → 边界不足）

### 尝试过的修复

```bash
# 1. 替换为系统自带的匹配 RGA 库
mv third_party/rga/lib/librga.so third_party/rga/lib/librga.so.bak
sudo cp /usr/lib/aarch64-linux-gnu/librga.so.2.1.0 third_party/rga/lib/librga.so
# → 换了系统版也一样，说明不是库版本问题

# 2. 放权 DMA heap 设备
sudo chmod 666 /dev/dma_heap/*
```

### 当前状态

RGA 硬件缩放偶尔失败时会回退到 CPU 软件处理，不影响数据通路。视频依然正常推流，只是偶尔卡顿。**属于可接受的优化项**，不是阻断问题。

### 长期解决方向

- 增加内核 CMA 预留大小（128MB → 256MB）
- 使用 `rga_lowmem.hpp` 的低内存 RGA 模式（项目中已有代码，但未默认启用）
- 或重编内核调整为更大 CMA 区域

# 第四部分：永久性环境修改

整个 debug 过程没有改过源码，但为了适配板子环境，做了以下**持久化变动**。如果重刷系统或换板子，这些操作需要重做。

## 1. 安装系统软件包

```bash
sudo apt update
sudo apt install cmake g++ make gdb -y        # C++ 编译 + 调试工具
sudo apt install openssh-server -y            # SSH 远程登录
```

| 包 | 用途 |
|----|------|
| cmake, g++, make | 编译项目 |
| gdb | 段错误时定位崩溃点 |
| openssh-server | PC 远程 SSH 到板子 |

## 2. 部署 rknn_server（NPU 用户态服务）

```bash
sudo cp /opt/.../third_party/rknn/bin/rknn_server → /usr/bin/rknn_server
sudo rknn_server &
```

**改了什么**：在 `/usr/bin/` 下新增了一个文件 `rknn_server`。

**为什么**：NPU 推理必须通过这个后台服务调度，项目 third_party 里有但没有自动部署。

**持久性**：每次开机需要重新启动 `sudo rknn_server &`（二进制文件已永久安装，只是进程不会自启动）。

## 3. 补 MPP 库版本符号链接

```bash
cd /opt/.../third_party/mpp/lib/
sudo ln -sf librockchip_mpp.so → librockchip_mpp.so.1
sudo ln -sf librockchip_mpp.so → librockchip_mpp.so.0
```

**改了什么**：在 `third_party/mpp/lib/` 下新增了两个符号链接（.so.1 和 .so.0 → .so）。

**为什么**：项目编译的二进制查找 `librockchip_mpp.so.1`，但 third_party 里文件名不带版本后缀，动态链接器找不到就跳到系统旧版 → API 不兼容 → 段错误。

**持久性**：永久文件，不重刷系统就一直在。

## 4. 替换 RGA 库为系统匹配版本

```bash
mv /opt/.../third_party/rga/lib/librga.so → librga.so.bak         # 备份原始
sudo cp /usr/lib/aarch64-linux-gnu/librga.so.2.1.0 → /opt/.../third_party/rga/lib/librga.so  # 替换
```

**改了什么**：`third_party/rga/lib/librga.so` 被替换为系统自带的 `/usr/lib/aarch64-linux-gnu/librga.so.2.1.0`。

**为什么**：third_party 带的 RGA 库版本（1.10.1）跟板子内核 RGA 驱动 ioctl 接口不完全匹配，导致 DMA 物理地址分配失败。系统自带的版本跟内核驱动是同一套 BSP 编译的，版本匹配。

**持久性**：永久替换。原始文件备份为 `librga.so.bak` 可供恢复。

## 5. 放权 DMA heap 设备节点

```bash
sudo chmod 666 /dev/dma_heap/cma /dev/dma_heap/cma-uncached /dev/dma_heap/system*
```

**改了什么**：让所有用户可读写 `/dev/dma_heap/*`（默认仅 root）。

**为什么**：RGA 硬件缩放需要从 DMA heap 申请物理连续内存，非 root 程序无权限访问。

**持久性**：**不持久！** 每次重启后 `/dev/` 下的权限会被 udev 重置，需要重新执行。长期方案是写 udev 规则，但因为项目中部分 RGA 调用不依赖 DMA heap 也能跑（CPU 回退），暂未配置。
